# Swarm Brain — инструкции для Claude

## Для Claude Code (разработка)

**Рабочая ветка: `sandbox_vas`** — вся разработка ведётся только здесь. Никогда не коммитить напрямую в `main`.

### Обязательно перед любым изменением кода

1. **Прочитай `QUICK_REF.md`** — компактная шпаргалка: деплой, ключевые файлы, callback-префиксы, сессии, env-переменные. За деталями — **`ARCHITECTURE.md`** (полное описание флоу, таблиц, структур).
2. После изменения — **сразу обнови `ARCHITECTURE.md`**: если поменялся флоу, callback, сессия, таблица или структура файлов — отразить в документе.
3. После каждой новой фичи или изменения функционала — сразу добавлять запись в **`CHANGELOG.md`**. Не копить в конце сессии, а писать по факту пока контекст свежий.
4. **После каждого завершённого изменения — сразу делать коммит** в `sandbox_vas`. Не накапливать несколько изменений в одном коммите. Коммит = одно логически завершённое изменение. Если сессия длится несколько часов — коммитить минимум раз в день, даже если работа ещё не закончена (WIP-коммит).

### Когда обновлять остальные документы

- **`README.md`** — обновлять при изменении технического стека, публичных эндпоинтов, способа развёртывания или цели проекта. Описание должно соответствовать тому, что реально работает.
- **`BACKLOG.md`** — обновлять при закрытии задачи (убрать или пометить done), появлении нового технического долга или идеи, изменении приоритетов. Не копить — писать сразу.
- **`SETUP_CLAUDE_DESKTOP.md`** — обновлять при изменении URL эндпоинтов, добавлении новых MCP-инструментов или изменении инструкций для Claude Desktop. Это инструкция для людей — она должна работать буквально.

**Правила:** Все `.md` файлы в корне проекта должны отражать текущее состояние системы. Если что-то изменилось и документ не обновлён — это незавершённая задача. Незакоммиченные изменения в конце сессии — тоже незавершённая задача.

## Безопасность данных — жёсткие правила

### Миграции БД
- `ADD COLUMN` — безопасно, делать сразу
- `DROP COLUMN`, `RENAME COLUMN`, `ALTER TYPE` — **только в два шага**: сначала убрать из кода → деплой → потом менять схему
- Никогда не писать `DELETE FROM` / `UPDATE` без `WHERE` — даже в миграциях
- Перед любой миграцией на prod — проверить на staging-окружении

### Код
- `SERVICE_ROLE_KEY` используется везде — RLS не защищает. Любой запрос к `supabase` идёт без ограничений. Это значит: **вся проверка доступа — только через код** (`checkAllowed`, `visibilityFilter`). Не обходить эти проверки, не добавлять прямые запросы в обход них
- При операциях удаления (`entries`, `tasks`) — всегда проверять что `userId` является владельцем или имеет право на удаление
- Cron-триггеры (`digest_cron`, `readai_token_refresh`, `setup_commands`) не аутентифицированы — не добавлять в них деструктивные операции
- `ADMIN_USER_ID = 744230399` зашит в `lib/supabase.ts` — если меняется, обновить там

### Известные архитектурные риски (не чинить без обсуждения, просто знать)
- Файлы в `swarm_drive` Storage отдаются по публичным URL без авторизации
- `requesting_user_id` в MCP передаётся вызывающим на доверии — нет JWT-верификации
- Granola API-ключи хранятся plaintext в `user_integrations`
- Начальная схема БД (`CREATE TABLE entries` и др.) не находится в `supabase/migrations/` — нельзя поднять проект с нуля

# Swarm Brain — инструкции для Claude Desktop

Ты работаешь с командной базой знаний Swarm Brain через MCP-инструменты.

## Главное правило

Когда тебе передают любой текст — транскрипт встречи, заметки, документ, переписку — **сначала сохрани оригинал** в базу знаний через `add_knowledge`, и только потом делай что-либо ещё. Не спрашивай разрешения, не предупреждай — просто сохраняй сразу.

## Порядок работы с транскриптом или заметками встречи

1. Сохрани полный оригинальный текст через `add_knowledge` (не сокращай, не редактируй)
2. Сделай краткое саммари: участники, ключевые темы, решения
3. Выдели задачи с ответственными и дедлайнами если есть

## Как называть секции в тезисах

Название секции должно отражать **тему**, а не конкретный инцидент или человека. Задай себе вопрос: **«Как бы коллега искал эту информацию через месяц, не зная деталей встречи?»** — и используй именно такое слово.

Называй широко: "Санитария" вместо "Дезинсекция", "Персонал" вместо имени сотрудника, "IT / Технические проблемы" вместо названия конкретной системы. Избегай имён собственных, аббревиатур и узкоспециальных терминов в заголовках — они могут быть непонятны тому, кто не был на встрече.

## Порядок работы с документом или файлом

1. Сохрани содержимое через `add_knowledge` с указанием источника (название файла)
2. Кратко опиши что в документе

## Поиск по базе знаний

Если спрашивают про встречи, задачи, решения, документы — используй `search_knowledge` или `get_tasks`, не отвечай по памяти.

## Что НЕ надо делать

- Не спрашивать "добавить в базу?" — добавлять сразу
- Не сокращать текст перед сохранением
- Не придумывать информацию которой нет в базе
