# ADR: Единый движок задач (_shared/tasks/)

**Дата:** 2026-05-31  
**Статус:** Принято и выполнено  
**Ветка:** `sandbox_vas`

---

## Контекст

Логика CRUD задач дублировалась в двух местах с расходящимся поведением:
- `swarm-bot/tasks/db.ts` — возвращает типизированные объекты, бросает исключения
- `swarm-mcp/tasks/tools.ts` — возвращает строки для Claude, inline-ошибки

Расхождения накапливались: разные лимиты, отсутствие `telegramId`-фильтра в MCP, разный порядок `nulls`.

---

## Решение

Выделить `_shared/tasks/{types,db}.ts` как единый движок с чистым контрактом.

**Что движок ДЕЛАЕТ:**
- Чистый CRUD к таблице `tasks`
- Принимает готовый `group_id` и готовых исполнителей
- Бросает исключение при ошибке
- `nullsFirst: false` везде, лимит дефолт 200

**Что движок НЕ ДЕЛАЕТ:**
- Не резолвит имя исполнителя → остаётся в прослойке MCP (`matchAssignee`)
- Не ищет workspace по telegram_id → остаётся в прослойке MCP (`resolveGroupId`)
- Не форматирует строки → остаётся в прослойке MCP

**Прослойки клиентов** держат всё клиент-специфичное:
- MCP: резолв user/assignee, catch→строка, форматирование, лимит 30
- Bot: тонкая обёртка, `dbListAllOpen` локально (другой sort)

---

## Сведение «случайных различий»

| Различие | Решение |
|----------|---------|
| Лимит 200 vs 30 | Параметр `limit` (дефолт 200); MCP передаёт 30 |
| `nullsFirst` | Явно `false` везде в движке |
| `telegramId`-фильтр | Добавлен в движок; доступен обоим клиентам |
| `select("id")` vs `select("*")` | Движок всегда `select("*")`; прослойка использует нужное поле |

---

## Порядок миграции

Три отдельных коммита по нарастанию риска:
1. `_shared/tasks/*` — создание, клиентов не трогаем
2. `swarm-mcp/tasks/tools.ts` — MCP на движок, `index.ts` не меняется
3. `swarm-bot/tasks/db.ts` — бот на движок, `handlers.ts` не меняется

---

## Известный остаток

Прямые `supabase.from("tasks")` в `handlers.ts` (~626,632,643) и `index.ts` (~326–327) — не трогались намеренно, следующий этап.
