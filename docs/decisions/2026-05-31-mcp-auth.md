# ADR: MCP-аутентификация через персональный токен

**Дата:** 2026-05-31  
**Статус:** Принято и выполнено  
**Ветка:** `sandbox_vas`

---

## Контекст

`requesting_user_id` приходил в теле JSON-RPC запроса без верификации — любой, знающий URL функции, мог подставить чужой ID и получить данные чужого воркспейса.

---

## Решение

**Храним хеш, не токен.** sha256(token) в hex в колонке `allowed_users.claude_mcp_token_hash`. При утечке базы хеш бесполезен без brute-force.

**Токен в заголовке.** `Authorization: Bearer smcp_<uuid>` — Claude Desktop отправляет с каждым запросом. Ни в теле, ни в URL.

**Одна точка проверки.** В `swarm-mcp/index.ts` сразу после разбора тела, до dispatch. Покрывает все 14 инструментов разом без изменения каждого.

**Инжекция проверенного ID.** `args.requesting_user_id = verifiedTelegramId` перед dispatch — значение из тела запроса игнорируется. Движок `_shared/tasks/db.ts` и сами инструменты не знают о механизме токена.

---

## Мягкий → жёсткий переход через `MCP_AUTH_REQUIRED`

Не ломаем текущее подключение во время перехода:
- `false` / не задан: нет заголовка → работает по-старому (backward-compatible)
- `true`: нет валидного токена → -32001 Unauthorized

Порядок: выдать токен → прописать в Claude Desktop → проверить → `MCP_AUTH_REQUIRED=true`.

---

## Ключевые решения

| Вопрос | Решение |
|--------|---------|
| Хранить хеш или plaintext? | Хеш (sha256 hex) — утечка базы не раскрывает токен |
| Отдельная таблица или колонка? | Колонка в `allowed_users` — у нас 5 пользователей, ротация не нужна |
| Срок жизни? | Нет — отзыв = `UPDATE SET claude_mcp_token_hash = NULL` |
| Ротация? | Нет — повторный вызов `generate_mcp_token()` заменяет хеш |
| Генерация? | SQL-функция `generate_mcp_token(telegram_id)` — показывает plaintext один раз |
| Код ошибки? | -32001 (диапазон -32000..-32099, server error по JSON-RPC 2.0) |
