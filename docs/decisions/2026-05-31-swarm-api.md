# ADR: swarm-api — архитектурные решения

**Дата:** 2026-05-31  
**Статус:** Принято и выполнено  
**Ветка:** `sandbox_vas`

---

## Поллинг вместо Realtime

**Решение:** Mini App сам переспрашивает данные каждые N секунд.  
**Почему:** Realtime требует RLS на таблице tasks и Supabase Auth bridge (выдача JWT по initData). Это две дополнительные задачи без очевидной пользы на старте — для канбана из 20 задач поллинг каждые 10 секунд неотличим от realtime.  
**Что это снимает:** RLS, Auth bridge, изменения в БД.

---

## group_id только из проверенной личности

**Решение:** `group_id` резолвится из `telegram_id` (полученного из подписанного initData) через `allowed_users`. Из тела запроса не принимается никогда.  
**Почему:** Если брать `group_id` из тела, любой может подставить чужой workspace. Telegram initData подписана HMAC с BOT_TOKEN — подделать невозможно.

---

## Формат заголовка: `Authorization: tma <initData>`

**Решение:** Prefix `tma` (Telegram Mini App) вместо `Bearer`.  
**Почему:** `Bearer` подразумевает непрозрачный токен. `tma` явно сигнализирует тип credentials и оставляет `Bearer` свободным для будущих JWT.

---

## Новая функция, не расширение swarm-mcp

**Решение:** Отдельный `swarm-api` Edge Function.  
**Почему:** swarm-mcp — JSON-RPC сервер для Claude Desktop, без REST и без initData. Разные протоколы, разные клиенты. Смешивать нет смысла.

---

## Исполнитель через telegram_id, не имя

**Решение:** Фронтенд передаёт `assignee_telegram_id: number`, swarm-api резолвит в имя через `user_profiles`.  
**Почему:** В Telegram Mini App у фронтенда есть список пользователей с их telegram_id (из `/users`). Передавать ID надёжнее чем имя — нет fuzzy-матчинга, нет неоднозначности.
