# swarm-mcp — Фикс аутентификации

> Только описание и предложение. Никаких изменений кода до согласования.

---

## 1. Как сейчас работает подключение Claude Desktop

**Протокол:** JSON-RPC 2.0 по HTTP POST на Supabase Edge Function.

**Аутентификации нет вообще.** Функция:
- Не читает заголовок `Authorization` (он объявлен в CORS allowed-headers, но нигде не проверяется)
- Не валидирует JWT (флаг `--no-verify-jwt` при деплое отключает даже Supabase-шлюзовую проверку)
- Не требует никакого ключа или секрета

**Откуда берётся `requesting_user_id`:**  
Это поле в теле JSON-RPC запроса (`params.arguments.requesting_user_id`), прилетает от Claude Desktop как часть аргументов инструмента. Не из заголовка — из тела. Пример запроса:

```json
{
  "jsonrpc": "2.0",
  "method": "tools/call",
  "params": {
    "name": "get_tasks",
    "arguments": { "requesting_user_id": 744230399 }
  }
}
```

Функция получает это число, идёт в `allowed_users`, находит `group_id` — и всё. Никакой проверки что это число вообще принадлежит тому, кто сделал запрос.

**Текущая точка входа без защиты:**  
`swarm-mcp/index.ts`, строки 705–713 — парсинг тела запроса. После этой точки код сразу расходится по инструментам. Единое место для вставки проверки — именно здесь, сразу после `const { method, params, id } = body`.

---

## 2. Уязвимость

Любой, кто знает URL функции, может послать запрос с произвольным `requesting_user_id` и получить задачи, записи базы знаний, список пользователей любого воркспейса. Workspace-изоляция есть, но она защищает только от случайного смешения данных — не от намеренного перебора ID.

Для текущего использования (один человек, Claude Desktop под своим управлением) риск минимален. Для Multi-user MCP или раскрытого URL — критично.

## 3. Предлагаемое решение: персональный токен

### Идея

Каждому пользователю выдаётся уникальный секретный токен. Он прописывается в настройках подключения Claude Desktop один раз. При каждом запросе Claude Desktop передаёт токен в заголовке `Authorization: Bearer <token>`. Сервер верифицирует токен → получает `telegram_id` → дальше всё работает как сейчас, только `requesting_user_id` уже не нужно передавать вручную — он берётся из токена.

### Где хранить токены

**Вариант A (минимальный):** добавить колонку `claude_mcp_token TEXT UNIQUE` в таблицу `allowed_users`. Простота: одна миграция, один запрос для проверки.

**Вариант B (масштабируемый):** отдельная таблица `mcp_api_tokens` с полями `telegram_id`, `token`, `created_at`, `last_used_at`, `is_active`. Поддерживает ротацию токенов, несколько токенов на пользователя, отзыв.

Рекомендую **вариант A** на старте: у нас 5 пользователей, ротация не нужна, одна колонка — одна миграция, нулевая сложность.

### Что меняется в коде (одна точка)

В `swarm-mcp/index.ts`, сразу после строки `const { method, params, id } = body` (строка ~712):

1. Читаем заголовок `Authorization`
2. Если отсутствует или не `Bearer` → возвращаем JSON-RPC ошибку `-32601 Unauthorized`
3. Берём токен, ищем в `allowed_users` по колонке `claude_mcp_token`
4. Если не нашли → ошибка
5. Если нашли → получаем `telegram_id`, кладём в контекст
6. Дальше все инструменты используют этот `telegram_id` как `requesting_user_id` — не из args, а из проверенного токена

Это **одно место**, покрывает все инструменты разом. Не нужно трогать каждый из 14 tool-диспатчей.

### Генерация и выдача токенов

Способы:
- Суперадмин генерирует через `/superadmin` → новый пункт "MCP-токен" в деталях пользователя
- Или прямо в БД через Supabase Dashboard — токен прописывается вручную

Формат токена: `crypto.randomUUID()` — 36 символов, достаточно. Можно добавить префикс `smcp_` для читаемости в логах.

## 4. Что сломается при переходе?

**Сломается:** текущее рабочее подключение Claude Desktop — если проверку сделать обязательной.

**Чтобы ничего не отвалилось:**

**Этап 1 — мягкий переход (backward-compatible):**  
Добавить колонку `claude_mcp_token`, проверку делать только если заголовок `Authorization` **присутствует**. Если заголовка нет — работать как сейчас. Это позволяет:
- Сначала выдать токен и прописать в Claude Desktop
- Проверить что токен работает
- Только потом сделать проверку обязательной

**Этап 2 — жёсткий переход:**  
После того как у всех пользователей токены прописаны — сделать проверку обязательной.

**Порядок действий:**
1. Миграция: добавить `claude_mcp_token` в `allowed_users`
2. Сгенерировать токен для текущего пользователя (744230399), записать в БД
3. Деплой swarm-mcp с мягкой проверкой
4. Прописать токен в настройках Claude Desktop: `Authorization: Bearer <token>`
5. Проверить что всё работает
6. Деплой с жёсткой проверкой (Authorization обязателен)

На этапах 2–5 текущее подключение продолжает работать без изменений.
